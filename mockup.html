5. Security Considerations

API Key Storage Options

// Proposed encryption approach
1. User enters API key
2. Generate salt using crypto.getRandomValues()
3. Derive key using PBKDF2 with user's extension ID
4. Encrypt with AES-GCM via Web Crypto API
5. Store encrypted blob in browser.storage.local
6. Decrypt only in background script for API calls

// Never:
- Store plaintext keys
- Log keys to console
- Expose keys to content scripts
- Send keys to sidebar except masked (****4hJ9)

Session Storage Option

  • 🔓 Keys stored only in memory
  • ✅ Lost on browser restart
  • ✅ Maximum security
  • ❌ Must re-enter each session

Persistent Storage Option

  • 🔒 Keys encrypted and stored
  • ✅ Survives browser restart
  • ⚠️ Encrypted but still local
  • ✅ Convenient for users

Key Design Decisions:
• Start with simple sequential multi-model (Option A) - easier to implement
• Prioritize security before adding API providers
• Keep existing chat functionality working during transition
• Use dropdown for cleaner UI that scales better
• Allow both session and persistent key storage with clear indicators
